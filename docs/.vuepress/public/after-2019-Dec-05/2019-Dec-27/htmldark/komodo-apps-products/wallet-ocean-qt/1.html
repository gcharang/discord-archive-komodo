if we can hire 100 devs, we can have 100 CC devs
but devs dont work for free and this is where the 100s of millions raised in ICO makes a difference, they can and do hire hundreds of devs
with CC, it is harder to make than an ETH smart contract, but easier to do than a ETH smart contract to invoke an existing CC. and this using of the existing CC is far more what is the level of the less serious developer
from command line arguments, you will be able to create a pegged and backed chain using any crypto and pegged to anything that has a price API
think of it as writing bash scripts
not anything hardcore, yet a lot has been created with bash scripts.
as it is the handful of CC devs are making enough new tech to keep the marketing side saturated. if we had 100 CC devs... there would be way too much tech and the 1/tech effect will kill the price
saying that KMD is a closed system because it is open source but not using the most trendy languages is silly
you can write anything in one language that you can in another, i was just an simple C programmer, yet i seem to have been able to write a lot of relevant blockchain code.
there is a reason almost all the core cryptographic code is written in C
having the CC being based on C (C++) is not any disadvantage in the difficulty to create full featured CC, and nothing stops bindings in other languages to be made.
we have experience of making language bindings in many different languages with supernet_api, hardly anybody used these other languages. but nothing stops anybody from doing the same for the CC
tl:dr to have inexperienced weekend coders be able to create a CC in a weekend is not realistic (nor anything that should be desired), but in a weekend you can do "shell scripts" that invoke existing CC and that is 10x to 100x the programmers that can do things at that level
