blackjok3r 08-Jul-19 01:41 AM
https://github.com/KMDLabs/komodo/commit/be1721cb80349905334a7d0aceff9826329f40c6 For those who care, I wrote this last night before bed, was thinking about it all day while driving to the city. Went a back way that was about 1.5H longer lol, but it was a really nice drive though all this country with riversand hills. Really good day, Basically, how it works is this: 1) On the first withdraw an object is created that contains the winner, before any bets have been spent from the game address, and after a result can be calculated, we can use the withdraw RPC to calculate the winner, Then in validation the exact same function is called to verify that the correct winner was added into this object. 2) next withdraw and all withdraws after this, simply spend the withdraw vout, into a new vout with the same object. 3) The validation simply hashes this object from the vin, and the vout and compares they are the same. So we know that its is valid. 3) This means, that we can have 1 ticket with a fixed price if needed now, because even if there is say a million tickets have been sold or even a billion lol, the validation only needs to run once, and then the winner can just keep withdrawing over and over to merge the tickets and spend them to their specified pubkey. The overhead after the first withdraw is very low, and shouldnt slow all nodes down while the winner is extracting their funds. (edited)
